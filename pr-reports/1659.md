# PR 1659 Report: Add 6529 Help Bot V1 Runtime

## PR

- GitHub PR: https://github.com/6529-Collections/6529seize-backend/pull/1659
- Branch: `codex/6529-help-bot-spec`
- Target: `main`
- Related frontend PR: https://github.com/6529-Collections/6529seize-frontend/pull/2786

## Task Summary

Implement V1 of `@6529help` without full RAG: fast trigger detection, visible bot status reactions, reply drops, cached frontend help-index retrieval with canonical URLs, bounded public-data DB lookups, Bedrock wording with deterministic fallback, and reliable failure replies.

## Backend Change Summary

- Added a post-create drop hook that detects explicit `@6529help` mentions, mention-only replies to someone else's question, and direct replies to prior bot replies.
- The bot handle is hardcoded as `@6529help`; runtime resolves that handle to the current profile id before posting reactions or replies.
- Creating the `6529help` profile is the activation gate; there is no enable/profile/queue env var.
- Added `help_bot_interactions` for idempotency, status, auditability, and retry-safe worker processing.
- Added `target_drop_id` so a user can tag `@6529help` in a reply while the bot reacts and replies to the original question drop.
- Added `helpBotReplyLoop`, backed by SQS `help-bot-replies` plus DLQ, to answer asynchronously.
- Added cached frontend help-index retrieval from hardcoded `https://6529.io/help-index.json`.
- Cold index-load failures now surface as technical failures instead of no-reliable-source answers.
- Kept V1 retrieval deterministic with alias/keyword matching over the frontend-owned corpus.
- Added a bounded public-data query-intent mode for aggregate/public Meme Card and TDH questions such as SZN counts, Meme Card TDH rate, highest edition size, and total TDH.
- Bedrock selects a semantic plan from a hardcoded public-data catalog; Bedrock output never contains executable SQL, table names, columns, joins, or expressions.
- The backend public-data compiler validates the selected entity, operation, metric, numeric filters, and limit, then emits parameterized SQL that forces the read pool, applies hard row limits, and injects a MySQL execution-time hint from backend code.
- Added Bedrock rendering behind an internal renderer with a fixed short timeout; deterministic record answers remain the fallback when Bedrock fails or times out.
- Added internal bot reaction/drop writers for 👀, ✅, ⚠️, bot replies, no-reliable-source replies, optional tech-team mentions, and technical-failure replies.
- Updated deployment config generation and architecture/runtime docs for the new API-to-SQS-to-worker flow.

## Backend Impact Summary

- New deployable service: `helpBotReplyLoop`.
- Existing deployable service changed: `api`.
- DB/entity changes: new TypeORM entity/table `help_bot_interactions`.
- Queue changes: new SQS queue `help-bot-replies` and `help-bot-replies-dlq`.
- Public data reads: fixed backend query templates over `nfts`, `memes_extended_data`, and `latest_tdh_global_history` for V1 aggregate lookup questions.
- External integration: Amazon Bedrock invocation with a hardcoded backend model id and deterministic fallback.
- Public API contract changes: none.

## Env/Config Changes

- Required: none. If the `6529help` profile exists, the backend can resolve it
  and run; otherwise it no-ops.
- Optional: `HELP_BOT_TECH_TEAM_HANDLES`, a semicolon-separated array of handles
  appended as real mentions when the bot has no reliable answer.

Operational confirmations:

- `seizeAPI` currently runs as `arn:aws:iam::987989283142:role/lambda-vpc-role`, the same role covered by the `help-bot-replies` queue policy.
- AWS IAM simulation returns `allowed` for `lambda-vpc-role` invoking the hardcoded Bedrock model `anthropic.claude-3-sonnet-20240229-v1:0`.
- The backend mysql pools are created without `multipleStatements`, so the mysql driver default remains disabled; the help-bot public-data path does not accept executable SQL from Bedrock.
