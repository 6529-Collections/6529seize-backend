# PR Report: Wallet Auth Session V2 Hardening

PR: https://github.com/6529-Collections/6529seize-backend/pull/1616
Branch: `codex/wallet-security-hardening` -> `main`
Related PRs: https://github.com/6529-Collections/6529seize-frontend/pull/2556
Mode: `staging-deploy`

## Summary

Backend auth v2 hardening for wallet sessions, structured signatures, connection
sharing, legacy-refresh migration, and backend-owned rollout settings. This PR
is intended to deploy backend first as a silent release, then frontend, then
enable migration behavior through backend env changes.

## What Changed

- Added session-v2 web/native auth routes, refresh/logout behavior, and
  connection-share create/redeem support.
- Connection-share creation now requires a matching active session-v2 web
  cookie; legacy bearer/JWT auth alone cannot mint a connection share.
- Connection-share deep links now carry only the one-time code and address.
  Role remains server-side on the share record and is returned by redeem.
- Hardened structured wallet signature validation, replay protection,
  domain/audience checks, and websocket auth compatibility.
- Session-v2 nonce challenges now use the accepted request API host as the
  structured-signature audience, so staging signs `api.staging.6529.io`
  without a dedicated audience env override. If the host cannot be accepted,
  the backend falls back to `API_BASE_URL`, then `api.6529.io`.
- Bound legacy refresh-token role handling server-side and added
  `AUTH_LEGACY_REFRESH_DISABLED` for the final v1 refresh cutoff.
- Added host-aware credentialed CORS for cookie-backed web auth routes while
  keeping general public API CORS wildcard/non-credentialed for external API
  clients.
- Added `WEB_APP_ORIGIN` and `WEB_APP_ADDITIONAL_ORIGINS` as the source of
  truth for first-party web origins; `AUTH_WEB_CREDENTIAL_ORIGINS` remains only
  as a deprecated additive compatibility alias.
- Rewrote `docs/auth/session-v2-migration-plan.md` as a phase-based deployment
  runbook instead of an env-var dump.

## Config / Env

- `AUTH_SESSION_HASH_SECRET`: backend API, strongly recommended before rollout.
  Dedicated secret for hashing session cookies, native refresh tokens, and
  connection-share codes. Falls back to `JWT_SECRET` if unset. Set before any
  v2 sessions are created; rotating it later invalidates existing v2 web
  cookies, native refresh tokens, and unredeemed connection-share codes.
- `WEB_APP_ORIGIN`: backend API, recommended but not required for normal prod
  and staging. Production shape: `https://6529.io`; staging shape:
  `https://staging.6529.io`.
- `WEB_APP_ADDITIONAL_ORIGINS`: backend API, optional comma-separated additive
  list for preview or extra first-party web origins.
- `AUTH_WEB_CREDENTIAL_ORIGINS`: backend API, deprecated compatibility alias.
  Still accepted additively; do not use for new deploys.
- `AUTH_CONNECTION_SHARING_DISABLED`: backend API, optional kill switch.
  Missing/false means connection sharing is enabled by default.
- `AUTH_LEGACY_REFRESH_DISABLED`: backend API, final v1 refresh cutoff. Keep
  unset/false through silent rollout and grace period; set true only when
  `/auth/redeem-refresh-token` should return `410 Gone`.
- `AUTH_STRUCTURED_SIGNATURES_REQUIRED`: backend API/settings strict flag. Keep
  unset/false until web, native, and known external clients are verified.
- `SESSION_V2_MIGRATION_DEADLINE`: backend API/settings rollout flag. Keep unset
  for silent release; later set an ISO timestamp with timezone to start FE
  migration prompts without another FE deploy.
- `AUTH_LEGACY_WS_QUERY_TOKEN_ENABLED`: backend API websocket compatibility.
  Keep unset/true until older websocket clients have moved off query-token auth.
- `AUTH_SIGNATURE_ALLOWED_AUDIENCES`: backend API, optional additive list when
  an extra accepted API audience is needed. Normal production and staging do not
  need `AUTH_SIGNATURE_AUDIENCE`; session-v2 nonce audience is derived from the
  accepted request API host with `API_BASE_URL` as fallback.

Built-in web-origin defaults:

- `api.6529.io` allows credentialed web auth from `https://6529.io`.
- `api.staging.6529.io` allows credentialed web auth from
  `https://staging.6529.io`.
- Localhost API hosts allow common localhost frontend ports.
- Session-v2 signable messages served through `api.6529.io` and
  `api.staging.6529.io` use that API host as `Audience`.

## Release Notes

- Backend can be deployed first silently; existing v1 refresh sessions continue
  while strict mode and legacy-refresh shutdown remain off.
- Connection sharing is enabled by default unless
  `AUTH_CONNECTION_SHARING_DISABLED=true`.
- Creating a connection share requires active session-v2 web auth. Existing
  legacy users can still use v1 refresh during the grace period, but must
  update auth before sharing a connection.
- Frontend migration prompts are controlled by backend
  `SESSION_V2_MIGRATION_DEADLINE` after both backend and frontend are deployed.
- Production rollout order:
  backend env for silent rollout -> `dbMigrationsLoop` -> `api` -> frontend
  PR plus native client release readiness -> set
  `SESSION_V2_MIGRATION_DEADLINE` -> later enable strict signatures and
  legacy-refresh shutdown after web, native, known external-client compatibility
  checks, and client communication.

## Risks / Follow-Ups

  global setup; this workstation has no working container runtime, so focused
  no-DB auth tests were used for the auth-sensitive paths.
- Full local `tsc --noEmit` currently fails on unrelated existing
  `profile-cms`/`zod` errors outside this auth change.
- Do not set `AUTH_STRUCTURED_SIGNATURES_REQUIRED=true` or
  `AUTH_LEGACY_REFRESH_DISABLED=true` until the migration and external-client
  communication steps in `docs/auth/session-v2-migration-plan.md` are complete,
  including native client availability and verification.
