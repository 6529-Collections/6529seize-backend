# PR Report: Subscription Wave Notifications

PR: https://github.com/6529-Collections/6529seize-backend/pull/1734
Branch: codex/subscription-wave-notifications -> main
Related PRs: None

## Summary
Adds optional Wave posts alongside the existing subscription Discord notifications for the six subscription minting events already emitted by the backend loops.

## What Changed
- Added a subscription wave notifier that creates best-effort chat drops when subscription Wave configuration is present.
- Wired Wave posts into `subscriptionsDaily`, `subscriptionsTopUpLoop`, and `transactionsProcessingLoop` without changing the existing Discord sends.
- Added admin handle mentions for duplicate top-up and redemption warning/error cases.
- Resolves top-up wallets to profile handles when available and mentions the topping-up profile in the Wave post.
- Hides link previews on generated subscription Wave drops.
- Sends redemption Wave posts only after the subscription redemption DB transaction commits, and uses `SUBSCRIPTIONS_CHAIN_ID` for those transaction links.
- De-dupes duplicate already-processed top-up Wave warnings by transaction hash.

## Backend Impact
- New services: None
- Changed services: `subscriptionsDaily`, `subscriptionsTopUpLoop`, `transactionsProcessingLoop`
- Planned/deployed services: `subscriptionsDaily`, `subscriptionsTopUpLoop`, `transactionsProcessingLoop` queued for staging deploy from `1a-staging`; deploy completion not awaited by request
- API: Not affected
- DB/entities/migrations: Schema/entities/migrations not affected; Wave drop creation uses existing drop, mention, score-refresh, and push-notification writes
- Queues/events/integrations: Adds optional Wave drop creation and identity push notifications from existing subscription loops

## Config / Env
- `SUBSCRIPTIONS_WAVE_ID`: New optional backend runtime env. When present with `SUBSCRIPTIONS_BOT_PROFILE_ID`, subscription loops post to this Wave. If unset, Wave posting is skipped and Discord behavior remains unchanged.
- `SUBSCRIPTIONS_BOT_PROFILE_ID`: New optional backend runtime env. Identifies the bot profile author for subscription Wave drops. If unset or not found, Wave posting is skipped and Discord behavior remains unchanged.
- `SUBSCRIPTIONS_ADMIN_HANDLES`: New optional backend runtime env. Comma- or semicolon-separated profile handles to mention on duplicate top-up and redemption warning/error posts. If unset, those posts are still created without admin mentions.

## Backend Deployment
- Deployed API: None
- Staging branch: merged to `1a-staging`
- Staging deploy order: `subscriptionsDaily`, `subscriptionsTopUpLoop`, `transactionsProcessingLoop`
- Deploy completion: not awaited by request

## Release Notes
When configured, subscription minting notifications now appear as Wave posts in addition to Discord. Normal top-up posts can mention the profile tied to the topping-up wallet, and admin-oriented warning/error posts can mention configured subscription admin handles.

## Risks / Follow-Ups
- Operational setup is required before Wave posts appear: create/select the bot profile, create/select the Wave, grant the bot permission to post there, and configure the new env values for the affected Lambda services.
