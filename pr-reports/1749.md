# PR Report: Link Main Stage winners to Meme cards

PR: https://github.com/6529-Collections/6529seize-backend/pull/1749
Branch: `codex/main-stage-meme-links` -> `main`
Related frontend PR: https://github.com/6529-Collections/6529seize-frontend/pull/3261

## Summary

The existing drop and wave-decisions responses now identify the Meme card minted from a winning submission in The Memes Main Stage. Historical relationships are reconstructed from minting-claim anchors, while new minting claims persist the relationship transactionally.

## What Changed

- Add an optional Meme card ID to winner-drop persistence and API contexts.
- Create the nullable column and unique index explicitly before any backfill runs.
- Backfill Main Stage winners by validating claim anchors against the ordered winner sequence, then walking backward and forward from those anchors.
- Resolve the target wave from `MAIN_STAGE_WAVE_ID`, and clear any relationship outside that wave; a staging smoke test caught and corrected the unsafe alternative of inferring the wave from all claim anchors.
- Reject inconsistent anchors, duplicate decision winners, invalid IDs, and conflicting existing mappings instead of guessing.
- Persist the card ID when future Main Stage minting claims are created.
- Treat future winner-to-card assignments as write-once and fail if a different card ID is already stored.
- Restrict reads and writes to the configured Main Stage wave so no other wave exposes this relationship.
- Let the backfill migrations no-op safely in environments without `MAIN_STAGE_WAVE_ID` or historical claim anchors.
- Expose the optional ID only through the canonical V2 submission context; the legacy winning-context contract is unchanged.

## Backend Impact

- New services: None
- Changed services: `dbMigrationsLoop`, `claimsBuilder`, `api`
- Required service order: `dbMigrationsLoop` -> `claimsBuilder` -> `api`
- API: V2 drop and decision responses can include optional `submission_context.meme_card_id`; no new endpoint and no legacy API change.
- DB/entities/migrations: Adds a nullable, unique Meme card ID on winner drops and backfills Main Stage history.
- Queues/events/integrations: The existing claim-builder flow writes the mapping in its current database transaction.

## Config / Env

Uses the existing `MAIN_STAGE_WAVE_ID` configuration. No new environment variables.

## Release Notes

Main Stage winning submissions can now link directly to the Meme card they produced, including historical winners that predate minting claims.
