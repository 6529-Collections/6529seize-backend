# PR Report: Link Main Stage winners to Meme cards

PR: https://github.com/6529-Collections/6529seize-backend/pull/1749
Branch: `codex/main-stage-meme-links` -> `main`
Related frontend PR: https://github.com/6529-Collections/6529seize-frontend/pull/3261

## Summary

The existing drop and wave-decisions responses now identify the Meme card minted from a winning submission in The Memes Main Stage. Historical relationships are reconstructed from minting-claim anchors into a dedicated one-to-one mapping table, while new minting claims persist the relationship transactionally.

## What Changed

- Add a dedicated `meme_card_drop_mappings` table with unique Meme card and drop IDs instead of adding a Main Stage-only field to generic wave-winner rows.
- Backfill Main Stage winners by validating claim anchors against the ordered winner sequence, then walking backward and forward from those anchors.
- Resolve the target wave from `MAIN_STAGE_WAVE_ID`; no mapping rows are created for any other wave.
- Reject inconsistent anchors, duplicate decision winners, invalid IDs, and conflicting existing mappings instead of guessing.
- Persist the card ID when future Main Stage minting claims are created; manually queued claims for other drops continue without creating a mapping.
- Treat both sides of future winner-to-card assignments as write-once and translate either unique-key conflict into a descriptive domain error.
- Restrict reads and writes to the configured Main Stage wave so no other wave exposes this relationship.
- Let the backfill migration no-op safely in environments without `MAIN_STAGE_WAVE_ID` or historical claim anchors.
- Expose the optional ID only through the canonical V2 submission context; the legacy winning-context contract is unchanged.

## Backend Impact

- New services: None
- Changed services: `dbMigrationsLoop`, `claimsBuilder`, `api`
- Required service order: `dbMigrationsLoop` -> `claimsBuilder` -> `api`
- API: V2 drop and decision responses can include optional `submission_context.meme_card_id`; no new endpoint and no legacy API change.
- DB/entities/migrations: Adds a dedicated one-to-one mapping entity/table and backfills Main Stage history without changing the generic winner-drop schema.
- Queues/events/integrations: The existing claim-builder flow writes the mapping in its current database transaction.
- Transaction/order guarantees: Runtime mapping writes reject callers without a request transaction. `dbMigrationsLoop` initializes TypeORM with `syncEntities: true` before invoking `db-migrate`, so the mapping table exists before the backfill. The MySQL migration driver wraps the backfill in its migration transaction, and deployment order keeps `claimsBuilder` behind a successful migration run.

## Config / Env

Uses the existing `MAIN_STAGE_WAVE_ID` configuration. No new environment variables.

## Release Notes

Main Stage winning submissions can now link directly to the Meme card they produced, including historical winners that predate minting claims.

## Staging

- The earlier column-based implementation was deployed at backend staging merge `bc894288`; that implementation is superseded and must not be promoted to production.
- The mapping-table rework has not been deployed.

The new data migration then reconstructs the mapping table directly from Main Stage winners and minting-claim anchors, so no manual database cleanup is required.
- After that migration succeeds, deploy `claimsBuilder` and then `api`.
